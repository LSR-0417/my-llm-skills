# sop-sdlc

軟體開發與測試標準協作流程（SDLC SOP）。

這份文件定義本團隊在軟體開發生命週期中的各個階段、參與角色、會議目的及預期產出物，確保產品從需求發起到上線測試，皆能順暢且高品質地推進。

## 1. 目的

- 建立從需求成型到送測簽核的完整 SDLC 協作流程。
- 明確定義每個階段的會議目的、參與角色、預期產出物與交付基準。
- 讓 `PRD`、`User Story`、`SRS`、`Issue`、`SDD`、測試案例、`PR` 與 `Release Note` 之間保持可追溯關係。
- 作為本專案後續撰寫 role skill 與其他 SOP skill 的流程參考來源。

## 2. 應作流程

### 階段一：需求成型 (Requirement Gathering)

會議目的：理解客戶（或市場）需求，釐清產品痛點與期望達成的商業價值。

主要參與者：客戶 / 業務代表、產品經理 (`PM / PO`)。

預期產出物：

- `PRD`（產品需求文件）：描述產品背景、目標客群、功能範圍與初步的 UI/UX 規劃。
- `User Story`（使用者故事）：以使用者視角撰寫的功能需求，必須包含明確的 `AC (Acceptance Criteria)`。

### 階段二：需求梳理與衝刺規劃 (Backlog Refinement & Sprint Planning)

會議目的：在正式進入開發前，針對 PM 提出的 `User Story` 進行技術可行性評估，擋掉不切實際的需求，並估算可行項目的開發時程。

主要參與者：`PM`、架構師、`RD Leader`（或資深 `RD`）。

預期產出物：

- `Sprint Backlog`：經過技術團隊確認可行，且評估過點數（`Story Points`）或工時，預計在下一個 Sprint 執行的實作列表。
- 註：技術上不可行或需延後的需求，將退回給 `PM` 重新評估或放入技術債清單。

### 階段三：技術設計與測試規劃 (Design & Test Planning)

此階段由開發與測試團隊針對已排入 Sprint 的需求，分頭並進進行準備工作，並明確劃分測試權責。

### 3.1 開發端：系統架構、Docs-as-Code 與單元測試

為了避免傳統文件與實作脫鉤，本團隊全面採用 Docs-as-Code 策略。所有技術設計文件必須與程式碼放在同一處管理。

主要參與者：架構師、`Developer`。

預期產出物：

- `SDD`（高階系統設計）：由架構師主導，定義系統整體架構與模組劃分，以 Markdown 格式存於 Git Repo 的特定目錄中。
- 細部設計與版本權威（Single Source of Truth）：API 規格（如 `Swagger / OpenAPI`）、資料庫綱要與核心業務邏輯說明，皆須以 Markdown 或註解形式，與程式碼一同存放在 Git Repository 中。
- 單元測試（`Unit Test`）與程式碼即文件：`Developer` 必須針對核心商業邏輯與演算法撰寫 `Unit Test`。透過 Clean Code 原則與測試案例來解釋邏輯，讓程式碼自己說話。`Unit Test` 的覆蓋率將作為 Code Review 的重要指標。
- 同步更新的程式碼與文檔：任何程式碼的變動，`Developer` 必須在發起 `PR (Pull Request)` 時同步更新對應的文檔。Reviewer 須一併審查「文檔與程式碼是否同步」。
- 自動化對外文件：針對需提供給外部部門（如 PM、客戶）的說明文檔，團隊將透過 `CI/CD` 流水線，自動將 Git Repo 中的 Markdown / API Spec 轉換為 `PDF` 或靜態網頁（如 `Docusaurus` / `GitBook`）並發布。

### 3.2 測試端：測試案例設計與交付 (Shift-Left Testing & BDD 精神)

`TE` 在此階段需提前產出測試劇本，並提供給 `Developer` 作為開發與自測的考卷標準。

主要參與者：`TE (Testing Engineer)`。

預期產出物（Test Cases 分類）：

#### Spec. Test（規格驗證測試）

- 定義：對應 PM 提供的 `User Story` 與 `AC`，代表最標準的 Happy Path。
- 目的：這是功能的最低及格線，提早交給 `RD`，讓 `RD` 根據此規格進行開發與自測。

#### Scenario Test（情境探索測試）

- 定義：`TE` 設想不在 Happy Path 內，但真實情境中有可能會發生，且數量與頻率不低的操作路徑（Alternate Path）或常見的使用者失誤，例如忘記密碼重置流程、漏填必填欄位等。
- 目的：用於確保系統在面對真實世界中各種常見的使用者行為時，能給出適當且友善的 UI/UX 引導與錯誤提示。

#### Edge Test（邊緣 / 極端測試）

- 定義：針對系統邊界條件、極端數值、併發衝突，或是惡意駭客攻擊（例如在年齡欄位故意輸入負數或超過 150 的無效數值、`SQL Injection` 攻擊等）所設計的測試。
- 目的：挖掘深層潛在風險與防禦漏洞，確保系統在遭受異常操作或惡意攻擊時的絕對強健性（Robustness），死守底線避免無預警崩潰或資料外洩。

### 階段四：開發實作與交付送測 (Development & Testing)

此階段由 `Developer` 完成功能開發並透過 `PR` 併入主線，隨後正式交付給 `TE` 團隊進行全面測試。為確保權責清晰，以下依據雙方視角定義標準動作。

### 4.1 開發端：開發、自測與交付 (Developer 視角)

主要參與者：`Developer`。

標準動作與防線：

#### 開發、自測與 PR 合併防線 (RD 守則)

- RD 的基本底線（送測門檻）：`Developer` 在準備將程式碼併入主要分支前，必須確保自己寫的 `Unit Test` 綠燈通過，且在本地端跑 `TE` 提供的 `Spec. Test` 沒壞。
- `CI/CD` 自動攔截（第一道防線）：發起 `PR` 時，Git 流水線將自動執行靜態程式碼掃描（Linting）與全域 `Unit Test`。若有任何紅燈或覆蓋率不達標，`PR` 將被鎖死（Block）不允許合併。

#### 部署測試環境與交接

- `PR` 成功合併後，`CI/CD` 將自動把主要分支部署至測試環境（`Test Env`）。
- `Developer` 正式交接給 `TE` 時，必須提供簡要的 `Release Note` 或操作說明。其中必須包含以下資訊：
- 對應的 `Spec. Test` 編號與版本（例如：參照 `Spec-Login-v1.2` 測試規格）。
- `Spec. Test` 自測通過之結果（例如：打勾確認或附上本地端自測成功截圖）。
- 測試所需的前置條件（例如：前置資料準備設定、特定的 VIP 測試帳號等）。

### 4.2 測試端：執行測試與回報 (TE 視角)

主要參與者：`TE (Testing Engineer)`。

標準動作與防線：

#### 執行測試 (TE 守則)

- `TE` 接收到 `Developer` 的 `Release Note` 並確認 `Test Env` 部署完成後，開始進行全面驗證。
- 依據階段三產出的完整 Test Case 進行測試，包含 `Spec. Test`（驗證基本規格）、`Scenario Test`（驗證情境與錯誤處理）與 `Edge Test`（防禦性極端驗證）。

#### 回報與驗收

- 發現問題：若發現 Bug、UI 跑版或狀態顯示異常，`TE` 須建立 `Issue`（附上重現步驟與截圖 / 錄影），並退回給 `Developer` 修復。修復後須重新走一次送測流程。
- 測試通過：若所有層級的測試皆通過，`TE` 給予 `Sign-off`（簽核），該功能正式達到 Done 的狀態，準備進入 Release 階段。

## 3. 禁止項目

- 不要在需求尚未收斂或 `User Story` / `AC` 尚未明確時直接開始正式實作。
- 不要在技術可行性與時程未評估前，就把需求直接排入開發。
- 不要讓設計文件、API 規格、資料結構說明與實作程式碼脫鉤。
- 不要在發起 `PR` 時漏掉對應文檔更新。
- 不要在送測前跳過 `Unit Test` 與 `Spec. Test` 的最低自測門檻。
- 不要在測試發現問題後只口頭通知，不建立可追蹤的 `Issue`。

#### 附件：如何評估點數 (Story Points)

`Story Points` 是一種在敏捷開發中用來衡量 `User Story` 相對投入程度的單位。它不是絕對時間（如小時或天），而是綜合考量多種因素後的估算值。

##### 1. 評估的三大核心要素

在估算點數時，開發團隊應同時考量以下三個面向：

- 複雜度（Complexity）：這個功能的邏輯有多複雜？需要處理很多關聯資料表或外部 API 嗎？
- 工作量（Volume of Work）：即使邏輯很簡單，但如果要處理大量重複性實作，例如 20 個類似的表單欄位，依然需要花費大量時間。
- 不確定性與風險（Uncertainty & Risk）：這是一項新技術嗎？需求夠不夠明確？有沒有潛在地雷？不確定性越高，點數應給予適度緩衝。

##### 2. 評估基準：費氏數列 (Fibonacci Sequence)

實務上通常使用費氏數列的變體來給予點數：`1, 2, 3, 5, 8, 13, 21...`

數字越大，代表跳距越大，這反映了「任務越大，估算越不精準」的現實。

- `1` 點：微小修改，例如改文案、調整按鈕顏色。
- `2 ~ 3` 點：熟悉且明確的小型功能開發。
- `5 ~ 8` 點：中大型功能，需要跨模組處理，或帶有一定的不確定性。
- `13` 點以上：通常代表這個 Story 太過龐大或風險極高。實務建議是要求 `PM` 將其拆解（Split）成更小的 Story 再進行開發。

##### 3. 實務作法：計畫撲克牌 (Planning Poker)

為了避免資深工程師的意見主導一切，團隊可以採用「計畫撲克牌」的形式：

- `PM` 說明完 `User Story` 與 `AC` 後，先釐清所有技術疑問。
- 團隊內所有參與開發的成員（包含 `Developer` 與 `TE`）同時亮出自己心中估算的點數。
- 如果大家點數一致（例如都評為 `5` 點），則直接定案。
- 如果點數差異過大（例如有人出 `2`，有人出 `8`），出最高與最低分的人必須說明原因。可能有人想到潛在風險，也可能有人認為可直接套用既有模組或套件。討論後再次投票，直到取得團隊共識。
