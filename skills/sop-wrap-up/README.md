# sop-wrap-up

這份 README 說明發完 PR 後的收尾原則。它只保留高層級清理流程與限制，避免收尾變成另一個不受控的破壞階段。

## 1. 目的

- 在 PR 建立後，整理這次開發對話留下的副產物。
- 清理這次主題明確產生的 worktree、branch、程序與暫存物。
- 保留仍有 ownership、未合併或狀態不明的內容。
- 讓對話結束前的工作區回到可理解、可交接、可安全封存的狀態。

## 2. 應作流程

- 先盤點這次主題建立或啟動了哪些 worktree、branch、程序、未追蹤檔案與暫存物。
- 關閉這次開發過程中啟動的測試伺服器或其他臨時程序。
- 檢查這次主題建立的 worktree 是否可安全移除，能移除時就清理。
- 只清理 ownership 明確且可安全移除的項目。
- 清理後再驗證目前還剩什麼，避免留下誤判。
- 列出這次實際做了哪些收尾動作，回報至少包含 `Stopped`、`Removed`、`Left intact` 與 `Remaining`。
- 最後判斷這個對話是否已可安全封存；若仍有未清狀態，要明確說明原因。

## 3. 禁止項目

- 不要刪除來源不明或 ownership 不清楚的東西。
- 不要刪除仍有未提交修改的 worktree 或尚未合併的 branch。
- 不要把使用者自己的修改或未知程序當成這次任務的副產物直接清掉。
- 不要在未確認未追蹤檔案、測試伺服器與 worktree 狀態前，就宣告可以安全封存對話。
- 不要跳過清理後驗證與回報。
